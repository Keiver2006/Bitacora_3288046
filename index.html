Conceptos de Tic

1. Algoritmo

Conjunto de pasos ordenados y finitos que permiten resolver un problema o ejecutar una tarea de manera sistemática.

2. Lenguaje de programación

Herramienta formal que permite comunicarse con la computadora para crear programas y sistemas.

3. Compilador

Programa que traduce el código fuente escrito en un lenguaje de alto nivel a lenguaje máquina.

4. Intérprete

Ejecuta el código línea por línea sin necesidad de compilarlo previamente.

5. Código fuente

Instrucciones escritas por un programador en un lenguaje comprensible para humanos.

6. Código binario

Representación del programa en ceros y unos, entendida directamente por el procesador.

7. Variable

Espacio en memoria que almacena un valor que puede cambiar durante la ejecución del programa.

8. Constante

Valor definido en el programa que no cambia en el tiempo de ejecución.

9. Estructura de control

Instrucciones que permiten decidir el flujo del programa, como condicionales o bucles.

10. Condicional if

Permite ejecutar una sección de código solo si se cumple una condición lógica.

11. Condicional else

Define qué hacer cuando la condición principal no se cumple.

12. Bucle for

Estructura que repite instrucciones un número definido de veces.

13. Bucle while

Repite instrucciones mientras se cumpla una condición lógica.

14. Función

Bloque de código reutilizable que ejecuta una tarea específica y puede devolver un valor.

15. Procedimiento

Similar a una función, pero no devuelve valor, solo ejecuta instrucciones.

16. Parámetro

Dato que se pasa a una función para personalizar su ejecución.

17. Argumento

Valor concreto que se asigna a un parámetro al llamar a una función.

18. Retorno

Valor que una función entrega después de ejecutarse.

19. Recursividad

Técnica donde una función se llama a sí misma para resolver problemas repetitivos.

20. Arreglo

Conjunto ordenado de elementos del mismo tipo de datos almacenados en posiciones contiguas.

21. Matriz

Arreglo bidimensional usado para representar tablas o gráficos.

22. Cadena de caracteres

Secuencia de letras, números o símbolos almacenados como texto.

23. Booleano

Tipo de dato que solo puede tomar dos valores: verdadero o falso.

24. Entero

Tipo de dato numérico que almacena valores sin decimales.

25. Decimal o float

Tipo de dato numérico que admite decimales.

26. Casting

Conversión de un tipo de dato a otro dentro de un programa.

27. Estructura de datos

Forma de organizar y almacenar datos para facilitar su manipulación.

28. Pila (stack)

Estructura LIFO donde el último elemento en entrar es el primero en salir.

29. Cola (queue)

Estructura FIFO donde el primer elemento en entrar es el primero en salir.

30. Lista enlazada

Conjunto de nodos donde cada elemento apunta al siguiente.

31. Hash table

Estructura que permite almacenar y buscar datos rápidamente mediante claves únicas.

32. Árbol binario

Estructura jerárquica donde cada nodo tiene como máximo dos hijos.

33. Gráfico (graph)

Conjunto de nodos conectados por aristas que representan relaciones.

34. Clase

Plantilla que define las propiedades y métodos de un objeto en programación orientada a objetos.

35. Objeto

Instancia concreta de una clase que contiene atributos y comportamientos.

36. Atributo

Propiedad que caracteriza a un objeto, como su nombre o valor.

37. Método

Función asociada a un objeto que define su comportamiento.

38. Encapsulamiento

Principio que protege los datos internos de un objeto, limitando el acceso directo.

39. Herencia

Capacidad de una clase de heredar atributos y métodos de otra.

40. Polimorfismo

Habilidad de un método o función para comportarse de manera diferente según el contexto.

41. Abstracción

Principio que simplifica la complejidad mostrando solo lo esencial de un objeto.

42. Sobrecarga

Definición de múltiples versiones de un método con diferentes parámetros.

43. Sobreescritura

Redefinición de un método heredado en una clase hija.

44. Instanciación

Proceso de crear un objeto a partir de una clase.

45. Constructor

Método especial que inicializa un objeto al ser creado.

46. Destructor

Método que libera recursos cuando un objeto deja de existir.

47. Módulo

Archivo independiente que contiene funciones o clases reutilizables.

48. Biblioteca

Conjunto de módulos predefinidos que amplían las capacidades de un lenguaje.

49. Framework

Conjunto estructurado de herramientas y librerías para desarrollar aplicaciones de manera más rápida.

50. API (Interfaz de programación de aplicaciones)

Conjunto de reglas y funciones que permiten la comunicación entre sistemas.

51. IDE (Entorno de desarrollo integrado)

Software que reúne editor, compilador y depurador en una sola herramienta.

52. Depuración

Proceso de identificar y corregir errores en un programa.

53. Bug

Error o fallo en un programa que provoca un comportamiento no deseado.

54. Breakpoint

Punto de parada en el código para analizar la ejecución paso a paso.

55. Debugging step-by-step

Ejecución del código línea por línea para identificar fallos.

56. Testing

Conjunto de pruebas para verificar que un programa funciona correctamente.

57. Test unitario

Verifica que una función o módulo específico se comporta como se espera.

58. Test de integración

Evalúa la interacción entre varios módulos del sistema.

59. Test de sistema

Verifica que el software completo cumpla con los requisitos.

60. Test de aceptación

Confirmación final de que el software cumple con las expectativas del usuario.

61. Refactorización

Mejorar el código existente sin alterar su funcionalidad.

62. Documentación

Textos y guías que explican el uso y funcionamiento del código.

63. Pseudocódigo

Descripción informal y simplificada de un algoritmo antes de programarlo.

64. Diagramas de flujo

Representaciones gráficas de los pasos de un algoritmo.

65. Paradigma imperativo

Programación que describe cómo deben ejecutarse las instrucciones paso a paso.

66. Paradigma declarativo

Programación que se centra en el qué se quiere lograr, no en cómo hacerlo.

67. Paradigma funcional

Estilo basado en funciones matemáticas y en evitar estados mutables.

68. Paradigma lógico

Basado en reglas lógicas para deducir resultados, como en Prolog.

69. Lenguaje compilado

Necesita ser traducido a código máquina antes de ejecutarse, como C++.

70. Lenguaje interpretado

Se ejecuta línea por línea, como Python o JavaScript.

71. Lenguaje tipado estáticamente

Requiere declarar el tipo de las variables antes de usarlas.

72. Lenguaje tipado dinámicamente

El tipo de las variables se asigna en tiempo de ejecución.

73. Tipado fuerte

Restringe la conversión automática entre tipos de datos distintos.

74. Tipado débil

Permite conversiones automáticas entre tipos, incluso si no son exactas.

75. Compilación JIT (Just In Time)

Traduce el código en el momento de ejecutarlo para optimizar el rendimiento.

76. Bytecode

Código intermedio que se ejecuta en una máquina virtual, como en Java.

77. Máquina virtual

Software que simula un sistema operativo para ejecutar programas independientemente del hardware.

78. Garbage collector

Mecanismo que libera automáticamente memoria no utilizada.

79. Optimización de código

Técnicas para hacer que un programa se ejecute más rápido o use menos recursos.

80. Complejidad algorítmica

Medida del rendimiento de un algoritmo en tiempo y espacio.

81. Big O Notation

Notación matemática que describe el comportamiento de un algoritmo en su peor caso.

82. Big Ω Notation

Representa el mejor caso en el tiempo de ejecución de un algoritmo.

83. Big Θ Notation

Representa el caso promedio del rendimiento algorítmico.

84. Algoritmo de búsqueda lineal

Recorre uno a uno los elementos hasta encontrar el deseado.

85. Algoritmo de búsqueda binaria

Divide el conjunto en mitades para localizar un elemento de forma eficiente.

86. Ordenamiento burbuja

Algoritmo que intercambia pares de elementos adyacentes hasta ordenar la lista.

87. Ordenamiento rápido (quicksort)

Divide y conquista para ordenar listas de manera eficiente.

88. Ordenamiento por inserción

Coloca cada elemento en su posición correcta dentro de una lista ordenada.

89. Ordenamiento por selección

Busca el mínimo y lo coloca en la primera posición, repitiendo el proceso.

90. Algoritmo de Dijkstra

Encuentra la ruta más corta entre dos nodos en un grafo.

91. Algoritmo A*

Variante heurística para encontrar rutas más cortas en grafos.

92. Programación concurrente

Ejecución de múltiples procesos de forma intercalada.

93. Programación paralela

Ejecución simultánea de procesos en diferentes núcleos de CPU.

94. Hilo (thread)

Unidad de ejecución más pequeña dentro de un proceso.

95. Proceso

Programa en ejecución que ocupa recursos de CPU y memoria.

96. Sincronización

Coordinación entre procesos o hilos para evitar conflictos en recursos compartidos.

97. Deadlock

Situación donde dos procesos esperan indefinidamente por un recurso que el otro posee.

98. Race condition

Error que ocurre cuando el resultado depende del orden de ejecución de procesos.

99. Semáforo

Mecanismo que controla el acceso a recursos compartidos en programación concurrente.

100. Mutex (mutual exclusion)

Objeto que asegura que solo un proceso acceda a un recurso a la vez.

101. Programación orientada a objetos (POO)

Paradigma que organiza el software en torno a objetos que combinan datos y comportamientos.

102. Paradigma basado en objetos

Forma de diseñar sistemas considerando cada entidad como un objeto con propiedades y acciones.

103. Clase abstracta

Clase que no puede instanciarse directamente y sirve de base para otras clases.

104. Interfaz

Contrato que define métodos que una clase debe implementar sin especificar cómo.

105. Composición

Relación entre objetos donde uno se forma a partir de la combinación de otros.

106. Agregación

Relación entre objetos donde uno contiene a otro, pero ambos pueden existir por separado.

107. Dependencia

Cuando una clase necesita de otra para funcionar correctamente.

108. Asociación

Relación simple entre objetos sin jerarquía ni dependencia estricta.

109. Relación de herencia múltiple

Capacidad de una clase para heredar de más de una clase base.

110. Clase base

Clase de la que heredan otras, también llamada clase padre.

111. Clase derivada

Clase que extiende o hereda atributos y métodos de otra.

112. Objeto inmutable

Objeto cuyos valores no pueden cambiar después de ser creado.

113. Objeto mutable

Objeto cuyos atributos pueden modificarse durante la ejecución.

114. Método estático

Método que pertenece a la clase y no a una instancia específica.

115. Método de instancia

Método que necesita de un objeto particular para ejecutarse.

116. Sobrecarga de operadores

Definir cómo los operadores matemáticos o lógicos funcionan con objetos.

117. Método virtual

Método que puede redefinirse en una clase derivada.

118. Método abstracto

Método sin implementación que debe definirse en clases hijas.

119. Clase sellada

Clase que no puede ser heredada por otras.

120. Método final

Método que no puede ser sobreescrito en una clase derivada.

121. Propiedad (property)

Atributo de un objeto accesible de forma controlada mediante getters y setters.

122. Getter

Método que devuelve el valor de un atributo privado.

123. Setter

Método que establece o modifica el valor de un atributo privado.

124. Encadenamiento de métodos

Técnica que permite llamar varios métodos consecutivos en una misma línea.

125. Fluent interface

Estilo de diseño que facilita la lectura mediante encadenamiento de métodos.

126. Instancia singleton

Patrón que asegura que solo exista un objeto de una clase en todo el sistema.

127. Prototype pattern

Patrón que crea nuevos objetos copiando un prototipo existente.

128. Factory method

Patrón que delega la creación de objetos a un método especializado.

129. Abstract factory

Patrón que genera familias completas de objetos relacionados.

130. Builder pattern

Patrón que permite construir objetos complejos paso a paso.

131. Adapter pattern

Patrón que traduce la interfaz de una clase para que sea compatible con otra.

132. Bridge pattern

Patrón que separa una abstracción de su implementación para mayor flexibilidad.

133. Composite pattern

Patrón que organiza objetos en jerarquías tipo árbol para tratarlos de manera uniforme.

134. Decorator pattern

Patrón que añade funcionalidades adicionales a un objeto sin modificar su estructura.

135. Facade pattern

Patrón que proporciona una interfaz simplificada para sistemas complejos.

136. Flyweight pattern

Patrón que optimiza el uso de memoria reutilizando objetos compartidos.

137. Proxy pattern

Patrón que controla el acceso a un objeto mediante un intermediario.

138. Observer pattern

Patrón que notifica automáticamente a varios objetos cuando ocurre un cambio.

139. Strategy pattern

Patrón que permite cambiar algoritmos en tiempo de ejecución.

140. State pattern

Patrón que cambia el comportamiento de un objeto según su estado interno.

141. Template method

Patrón que define la estructura de un algoritmo y delega detalles en subclases.

142. Command pattern

Patrón que encapsula solicitudes como objetos para mayor flexibilidad.

143. Iterator pattern

Patrón que permite recorrer colecciones sin exponer su estructura interna.

144. Mediator pattern

Patrón que centraliza la comunicación entre varios objetos para reducir dependencias.

145. Memento pattern

Patrón que guarda y restaura el estado de un objeto.

146. Interpreter pattern

Patrón que evalúa expresiones de un lenguaje definido.

147. Chain of responsibility

Patrón que delega solicitudes a través de una cadena de objetos.

148. Dependency injection

Técnica que provee dependencias a un objeto desde fuera en lugar de crearlas dentro.

149. Inversión de control (IoC)

Principio donde el flujo de control del programa es gestionado por un contenedor.

150. Principio SOLID

Conjunto de cinco buenas prácticas para diseñar software orientado a objetos.

151. Principio de responsabilidad única

Una clase debe tener un único motivo de cambio.

152. Principio abierto/cerrado

Una clase debe estar abierta a extensión pero cerrada a modificación.

153. Principio de sustitución de Liskov

Una subclase debe poder reemplazar a su superclase sin alterar el sistema.

154. Principio de segregación de interfaces

Es mejor tener interfaces pequeñas y específicas que una muy general.

155. Principio de inversión de dependencias

Las clases deben depender de abstracciones y no de implementaciones concretas.

156. UML (Lenguaje de modelado unificado)

Estándar gráfico para representar estructuras y relaciones en sistemas.

157. Diagrama de clases

Representación UML que muestra clases, atributos y métodos.

158. Diagrama de secuencia

Representación UML que describe cómo interactúan los objetos a lo largo del tiempo.

159. Diagrama de casos de uso

Representa las interacciones entre actores y el sistema.

160. Diagrama de objetos

Muestra instancias específicas de clases en un momento dado.

161. Diagrama de estados

Representa los estados que un objeto puede tener y cómo cambia entre ellos.

162. Diagrama de componentes

Describe los módulos de software y sus relaciones.

163. Diagrama de despliegue

Representa cómo el software se distribuye en el hardware.

164. Análisis orientado a objetos

Proceso de entender un problema en términos de objetos.

165. Diseño orientado a objetos

Traducción del análisis en estructuras de software concretas.

166. Metaprogramación

Técnica que permite a un programa manipular su propio código.

167. Reflexión

Capacidad de un lenguaje para inspeccionar y modificar objetos en tiempo de ejecución.

168. Serialización

Proceso de convertir un objeto en una secuencia de bytes para almacenarlo o enviarlo.

169. Deserialización

Reconstrucción de un objeto a partir de una secuencia de bytes.

170. Persistencia de objetos

Técnica para almacenar objetos más allá del ciclo de vida de un programa.

171. ORM (Object Relational Mapping)

Mecanismo que relaciona objetos de programación con tablas de bases de datos.

172. DAO (Data Access Object)

Patrón que separa la lógica de negocio del acceso a datos.

173. DTO (Data Transfer Object)

Objeto diseñado para transportar datos sin lógica adicional.

174. VO (Value Object)

Objeto que representa un valor inmutable en el sistema.

175. Entity

Objeto que tiene una identidad única dentro de una aplicación.

176. Aggregate

Conjunto de objetos relacionados tratados como una unidad.

177. Service object

Objeto que contiene lógica de negocio sin pertenecer a una entidad.

178. Repository

Objeto que actúa como colección de entidades con operaciones de búsqueda.

179. Domain model

Representación del conocimiento del negocio en objetos de software.

180. DDD (Domain-Driven Design)

Enfoque de diseño centrado en el dominio del negocio y su modelo.

181. Bounded context

División de un dominio grande en contextos más pequeños y manejables.

182. Event sourcing

Técnica donde el estado de un objeto se reconstruye a partir de eventos almacenados.

183. CQRS (Command Query Responsibility Segregation)

Separación de operaciones de lectura y escritura en sistemas.

184. Microservicio orientado a objetos

Arquitectura donde cada microservicio maneja su propio modelo de dominio.

185. Patrón Active Record

Objeto que combina datos y operaciones de persistencia.

186. Patrón Lazy Loading

Técnica que retrasa la carga de datos hasta que se necesitan.

187. Patrón Eager Loading

Técnica que carga los datos relacionados de inmediato.

188. Encapsulación de dominio

Restricción de acceso a la lógica central de negocio.

189. Inyección de dependencias constructor

Dependencias entregadas al crear el objeto.

190. Inyección de dependencias setter

Dependencias entregadas mediante un método modificador.

191. Mock object

Objeto falso usado en pruebas para simular comportamientos.

192. Stub

Objeto simple usado en pruebas que devuelve respuestas predeterminadas.

193. Dummy object

Objeto sin comportamiento real que solo rellena parámetros en pruebas.

194. Spy

Objeto de prueba que registra interacciones para verificar comportamientos.

195. Fake object

Objeto de prueba que implementa lógicamente pero de manera simplificada una funcionalidad.

196. Prueba orientada a objetos

Verificación de clases, objetos y sus interacciones en un sistema.

197. Acoplamiento bajo

Cuando las clases dependen poco unas de otras, facilitando mantenimiento.

198. Cohesión alta

Cuando una clase cumple un propósito claro y relacionado, mejorando legibilidad.

199. Principio Tell, don’t ask

Recomienda que los objetos actúen en lugar de exponer demasiados datos.

200. Objetos anémicos

Mal diseño donde las clases solo contienen datos sin lógica de negocio.

201. Ciclo de vida del software

Conjunto de fases que sigue un programa desde su concepción hasta su retiro.

202. Análisis de requisitos

Etapa donde se identifican las necesidades del cliente para diseñar la solución.

203. Especificación funcional

Documento que describe lo que debe hacer el sistema desde la perspectiva del usuario.

204. Especificación técnica

Documento que describe cómo se implementarán los requisitos a nivel tecnológico.

205. Arquitectura de software

Estructura general que define los componentes y su interacción.

206. Modelo en cascada

Metodología de desarrollo lineal donde cada fase se completa antes de iniciar la siguiente.

207. Modelo incremental

Estrategia que construye el software en pequeñas entregas funcionales.

208. Modelo iterativo

Se basa en crear versiones sucesivas del software hasta lograr el producto final.

209. Metodología ágil

Enfoque flexible que prioriza entregas rápidas, colaboración y adaptabilidad.

210. Scrum

Marco ágil que organiza el trabajo en sprints con roles y reuniones definidos.

211. Kanban

Método visual de gestión de tareas basado en tableros y tarjetas.

212. Extreme Programming (XP)

Metodología ágil que enfatiza la calidad mediante programación en pareja y pruebas frecuentes.

213. DevOps

Cultura que integra desarrollo y operaciones para mejorar la entrega continua.

214. Integración continua

Práctica que fusiona cambios en el código frecuentemente para detectar errores temprano.

215. Entrega continua

Proceso automatizado que asegura que el software esté listo para desplegarse en cualquier momento.

216. Despliegue continuo

Automatización completa que publica el software en producción tras pasar las pruebas.

217. Repositorio de código

Lugar centralizado donde se almacena y gestiona el código fuente.

218. Control de versiones

Sistema que registra cambios en el código y permite volver a versiones anteriores.

219. Git

Herramienta distribuida de control de versiones ampliamente usada en proyectos de software.

220. GitHub

Plataforma basada en Git que permite colaboración, revisión de código y gestión de proyectos.

221. Branch (rama)

Línea paralela de desarrollo en un repositorio que no afecta al código principal.

222. Merge (fusión)

Acción de combinar cambios de una rama con otra.

223. Pull request

Solicitud para que los cambios de una rama sean revisados e integrados en el proyecto.

224. Commit

Registro de cambios realizados en un repositorio con un mensaje descriptivo.

225. Tag (etiqueta)

Marca en un repositorio que señala una versión específica del software.

226. Release

Versión empaquetada de un software lista para distribución.

227. Versionado semántico

Convención que asigna números de versión indicando cambios mayores, menores o parches.

228. Backlog de producto

Lista priorizada de tareas y requisitos pendientes de un proyecto.

229. Sprint backlog

Conjunto de tareas seleccionadas para completarse en un sprint ágil.

230. Historias de usuario

Descripción corta de una necesidad escrita desde la perspectiva del usuario final.

231. Criterios de aceptación

Condiciones que debe cumplir una historia de usuario para considerarse terminada.

232. Epic

Conjunto de historias de usuario relacionadas que forman una gran funcionalidad.

233. Task board

Tablero que organiza visualmente tareas en estados como “pendiente”, “en progreso” y “hecho”.

234. Daily stand-up

Reunión breve diaria en metodologías ágiles para sincronizar avances y obstáculos.

235. Retrospectiva

Reunión de equipo donde se reflexiona sobre el trabajo realizado y posibles mejoras.

236. Demo de sprint

Presentación de las funcionalidades desarrolladas al final de cada iteración.

237. MVP (Producto mínimo viable)

Versión básica de un software que incluye solo lo esencial para probar su valor.

238. Prototipo

Representación preliminar de un sistema que sirve para validar ideas.

239. Mockup

Diseño estático que muestra la apariencia visual de una aplicación.

240. Wireframe

Boceto esquemático de la estructura de una interfaz sin detalles gráficos.

241. UI (Interfaz de usuario)

Parte visible con la que interactúa el usuario en una aplicación.

242. UX (Experiencia de usuario)

Conjunto de percepciones y emociones del usuario al interactuar con el sistema.

243. Diseño centrado en el usuario

Metodología que pone las necesidades del usuario como eje del desarrollo.

244. Responsive design

Diseño de interfaces que se adaptan a diferentes tamaños de pantalla.

245. Accesibilidad digital

Característica que permite que personas con discapacidades puedan usar el software.

246. Testing de usabilidad

Evaluación que mide lo fácil e intuitiva que resulta una aplicación.

247. Mock service

Servicio simulado usado en pruebas cuando el real no está disponible.

248. Pruebas automatizadas

Scripts que verifican automáticamente el funcionamiento del software.

249. Pruebas manuales

Evaluaciones realizadas por personas que interactúan directamente con el sistema.

250. Pruebas de regresión

Verifican que nuevas modificaciones no rompan funcionalidades previas.

251. Pruebas de carga

Evalúan cómo se comporta el software bajo grandes volúmenes de usuarios o datos.

251. Pruebas de carga

Evalúan cómo se comporta el software bajo grandes volúmenes de usuarios o datos.

252. Pruebas de estrés

Analizan la respuesta del sistema en condiciones extremas.

253. Pruebas de seguridad

Buscan vulnerabilidades que puedan ser explotadas en el software.

254. Pruebas alfa

Evaluaciones internas realizadas por desarrolladores antes de liberar el producto.

255. Pruebas beta

Evaluaciones externas realizadas por usuarios reales antes del lanzamiento oficial.

256. Plan de pruebas

Documento que organiza qué, cómo y cuándo se harán las pruebas de software.

257. Gestión de incidencias

Proceso de registrar, clasificar y resolver errores detectados en un sistema.

258. Bug tracker

981. Edición digital

Proceso mediante el cual se modifican archivos multimedia (imágenes, videos, audios o textos) utilizando software especializado para mejorar su calidad o adaptarlos a un propósito específico.

982. Interfaz de usuario (UI)

Es el diseño visual con el que interactúa el usuario dentro de una aplicación de edición. Una buena interfaz facilita el acceso a herramientas y funciones.

983. Renderización

Etapa final en la edición de video o animación donde el proyecto se convierte en un archivo visualizable (por ejemplo, MP4 o MOV), combinando efectos, sonidos y transiciones.

984. Filtros

Efectos predeterminados que se aplican a imágenes o videos para modificar su apariencia, color, brillo o estilo artístico de manera rápida.

985. Capas (Layers)

Sistema que permite trabajar diferentes elementos (texto, imagen, efectos) de forma separada dentro de un mismo proyecto. Es fundamental en programas como Photoshop o GIMP.

986. Recorte (Crop)

Herramienta utilizada para eliminar partes no deseadas de una imagen o video, enfocando la atención en el área principal.

987. Línea de tiempo (Timeline)

En edición de video o audio, es la representación visual del proyecto donde se organizan los clips, efectos y transiciones a lo largo del tiempo.

988. Formato de archivo

Tipo de extensión que define cómo se guarda la información de un proyecto (por ejemplo: .PSD, .MP4, .WAV, .DOCX). Cada formato tiene características específicas.

989. Corrección de color

Proceso que ajusta el tono, saturación y brillo de imágenes o videos para mejorar su calidad o lograr una estética coherente.

990. Máscaras

Herramientas que permiten ocultar o mostrar solo partes específicas de una imagen o video, muy usadas para efectos visuales avanzados.

991. Exportación

Acción de guardar o convertir un proyecto editado en un formato final para su uso o publicación.

992. Edición no destructiva

Tipo de edición en la que los cambios no alteran el archivo original, permitiendo revertirlos o ajustarlos después sin pérdida de calidad.

993. Transiciones

Efectos visuales que suavizan el cambio entre dos clips de video o imágenes, como desvanecimientos o barridos.

994. Pistas de audio

En las aplicaciones de edición de video o sonido, son líneas donde se colocan los efectos de sonido, música o diálogos.

995. Resolución

Cantidad de píxeles que componen una imagen o video. Cuanto mayor sea la resolución, mayor será la calidad visual.

996. Editor de texto enriquecido

Tipo de aplicación que permite crear y modificar documentos con formato, como negritas, colores, tablas o imágenes (por ejemplo, Word o Google Docs).

997. Inteligencia artificial en edición

Uso de algoritmos que ayudan a automatizar tareas de edición como eliminar fondos, mejorar fotos o sincronizar audio y video.

998. Edición colaborativa

Función que permite que varios usuarios trabajen simultáneamente sobre el mismo proyecto en línea, como en Canva o Google Docs.

999. Curva de aprendizaje

Nivel de dificultad que presenta una aplicación de edición. Algunas son intuitivas y otras requieren tiempo y práctica, como Adobe Premiere o DaVinci Resolve.

1000. Compresión

Técnica que reduce el tamaño de los archivos multimedia para facilitar su almacenamiento o envío, sin afectar demasiado la calidad.
